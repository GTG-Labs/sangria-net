## Project Structure

```
sangria-net/
  backend/                    # Go/Fiber API server (port 8080)
    main.go                   # Entry point — Fiber app, auth middleware, route registration
    handlers/                 # API key management (create, list, revoke, authenticate)
    merchantPaymentHandler/   # x402-compatible payment route stubs
    dbEngine/                 # PostgreSQL data layer (pgx pool, models, queries)
  sdk/
    sdk-typescript/           # TypeScript merchant SDK (Express, Fastify, Hono adapters)
    sdk-python/               # Python merchant SDK (FastAPI decorator)
  dbSchema/                   # Drizzle ORM schema — source of truth for DB shape
  frontend/                   # Next.js documentation and marketing site
```

## Core Components

### 1. Go Backend (`backend/`)

A **Fiber v3** HTTP server listening on `:8080`. Connects to PostgreSQL via a pgx pool and integrates with **WorkOS** for user identity.

**Authentication — two independent middleware layers:**

| Middleware | Header | Used for |
| --- | --- | --- |
| `workosAuthMiddleware` | `Authorization: Bearer <jwt>` | User/session routes — validates WorkOS JWT via a cached JWKS endpoint |
| `apiKeyAuthMiddleware` | `Authorization: Bearer <sg_...>` or `X-API-Key` | Merchant routes — bcrypt-validates the hashed key stored in the database |

**API Routes:**

| Method | Path | Auth | Status |
| --- | --- | --- | --- |
| `GET` | `/` | None | Health check — returns `"Hello, Sangria!"` |
| `POST` | `/v1/generate-payment` | None | Returns an x402 v2 challenge JSON |
| `POST` | `/v1/settle-payment` | None | Stub — always returns `success: false` |
| `POST` | `/users` | WorkOS JWT | Upsert user record on login |
| `GET` | `/api-keys` | WorkOS JWT | List user's API keys |
| `POST` | `/api-keys` | WorkOS JWT | Create a new API key (max 10 per user) |
| `DELETE` | `/api-keys/:id` | WorkOS JWT | Revoke an API key |
| `GET` | `/merchant/profile` | API Key | Returns user info and key metadata |
| `POST` | `/facilitator/verify` | API Key | **501 Not Implemented** |
| `POST` | `/facilitator/settle` | API Key | **501 Not Implemented** |

**Key files:** `backend/main.go`, `backend/handlers/`, `backend/merchantPaymentHandler/payments.go`

### 2. Database Layer (`backend/dbEngine/`)

A **pgx v5** connection pool backed by **PostgreSQL**. The Go models mirror the Drizzle schema defined in `dbSchema/`.

Key operations:

- `UpsertUser` — create or update a user record keyed by WorkOS ID
- `GetUserByWorkosID` — look up a user from their WorkOS identity
- `InsertTransaction` — double-entry ledger write for credit accounting
- API key operations — `CreateAPIKey`, `GetAPIKeysByUserID`, `RevokeAPIKey`, `AuthenticateAPIKey` (bcrypt verify + last-used timestamp update)

**Key files:** `backend/dbEngine/engine.go`, `backend/dbEngine/models.go`, `dbSchema/schema.ts`

### 3. TypeScript Merchant SDK (`sdk/sdk-typescript/`)

A framework-agnostic SDK that merchants drop into their Node.js API to require payment on any endpoint.

**Core flow** (`core.ts`):

1. If the incoming request has no `PAYMENT-SIGNATURE` header, call `POST /v1/generate-payment` on the Sangria backend and return an HTTP `402` with the x402 challenge as the response body.
2. If the header is present, call `POST /v1/settle-payment`. On success, attach paid metadata to the request and pass through. On failure, return a `402`.

**Framework adapters** — each reads the `PAYMENT-SIGNATURE` header and resource URL then delegates to core:

| Adapter | File |
| --- | --- |
| Express | `adapters/express.ts` |
| Fastify | `adapters/fastify.ts` |
| Hono | `adapters/hono.ts` |

**Example usage (Express):**

```typescript
import { SangriaNet } from "@sangria/sdk";
import { fixedPrice } from "@sangria/sdk/adapters/express";

const sangria = new SangriaNet({ apiKey: process.env.SANGRIA_API_KEY });

app.get("/premium", fixedPrice(sangria, { price: 0.001 }), (req, res) => {
  res.json({ data: "paid content" });
});
```

**Key files:** `sdk/sdk-typescript/src/core.ts`, `sdk/sdk-typescript/src/types.ts`, `sdk/sdk-typescript/src/adapters/`

### 4. Python Merchant SDK (`sdk/python/`)

A FastAPI-focused SDK with the same backend-call pattern as the TypeScript SDK.

**`require_sangria_payment` decorator** wraps a FastAPI route handler to:

1. Check for `PAYMENT-SIGNATURE` header. If absent, call `/v1/generate-payment` and return a `402` with a `PAYMENT-REQUIRED` response header (base64-encoded challenge).
2. If present, call `/v1/settle-payment`. On success, store the `SettlementResult` in `request.state.sangria_verification` and invoke the route. On failure, return `403`.

**`SangriaMerchantClient`** handles HTTP communication with the backend (configurable base URL and API key via `Authorization: Bearer`).

**Example usage:**

```python
from sangria_sdk import SangriaMerchantClient, require_sangria_payment
from decimal import Decimal

client = SangriaMerchantClient(api_key="sg_live_...", base_url="https://api.sangria.net")

@app.get("/premium")
@require_sangria_payment(client, amount=Decimal("0.001"))
async def premium(request: Request):
    return {"data": "paid content"}
```

**Key files:** `sdk/python/src/sangria_sdk/fastapi.py`, `sdk/python/src/sangria_sdk/client.py`, `sdk/python/src/sangria_sdk/models.py`

## Payment Route Flow

How a merchant endpoint interacts with the Sangria backend when a request comes in:

```
Client                  Merchant Server (SDK)       Sangria Backend (:8080)
  │                            │                            │
  │  GET /premium              │                            │
  │  (no PAYMENT-SIGNATURE)    │                            │
  │───────────────────────────>│                            │
  │                            │  POST /v1/generate-payment │
  │                            │  { amount, resource }      │
  │                            │───────────────────────────>│
  │                            │  x402 v2 challenge JSON    │
  │                            │<───────────────────────────│
  │  HTTP 402                  │                            │
  │  PAYMENT-REQUIRED: <b64>   │                            │
  │<───────────────────────────│                            │
  │                            │                            │
  │  GET /premium              │                            │
  │  PAYMENT-SIGNATURE: <sig>  │                            │
  │───────────────────────────>│                            │
  │                            │  POST /v1/settle-payment   │
  │                            │  { paymentHeader, ... }    │
  │                            │───────────────────────────>│
  │                            │  { success: true, tx }     │
  │                            │<───────────────────────────│
  │  HTTP 200 + data           │                            │
  │<───────────────────────────│                            │
```
